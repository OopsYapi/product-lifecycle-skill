# 阶段2：方案设计（至少2种方案）

> 好的产品设计不是"设计一个方案"，而是"设计至少2种方案，然后做出选择"。本阶段目标：**系统性地探索多种可能性，通过对比找到最优解**。

---

## AI 执行指引

进入本阶段时，按以下步骤操作：

1. **读取前置产物**：读取 `01-discovery/` 下的需求判别报告，确认真实需求
2. **逐步引导提问**：按下方「引导提问」分3个部分依次讨论
3. **确保至少2种方案**：若用户只提出1种方案，主动设计1种对比方案（从不同设计理念出发）
4. **生成方案设计文档**：按「输出模板」生成
5. **触发审核**：加载 `references/reviewers/reviewer-tech.md`，以技术视角审核方案可行性
6. **归档**：将文档保存到 `~/.workbuddy/products/{产品名称}/02-solutions/{产品名称}-方案设计.md`
7. **判断下一步**：
   - 若需做竞品分析 → 进入 **阶段3：竞品分析**
   - 若不需要竞品分析 → 进入 **阶段4：商业模式 & 盈利设计**

---

## 本阶段目标

输出：
- **至少2种解决方案**（方案A、方案B，必要时方案C）
- **方案对比矩阵**（成本/收益/风险/战略对齐度）
- **推荐方案 + 推荐理由**
- **MVP范围定义**

---

## 引导提问

### 第一部分：约束条件

1. **约束条件有哪些？**

   | 约束类型 | 需明确的内容 |
   |---------|-------------|
   | 技术约束 | 现有架构、技术栈、性能要求 |
   | 资源约束 | 开发人力、时间、预算 |
   | 业务约束 | 商业模式、收费模式、合作模式 |
   | 合规约束 | 法律、隐私、行业规定 |
   | 用户约束 | 用户习惯、学习成本、使用场景 |

### 第二部分：方案探索

2. **方案设计的核心理念是什么？**

   > 每个方案应有清晰的"设计哲学"，而不是零散的功能堆砌。

   **常见设计理念对比**：
   - "轻量介入，最小化用户摩擦" —— 能不做的就不做，能简化的就简化
   - "完整覆盖，一网打尽" —— 覆盖所有场景，不留死角
   - "分阶段推进，先验证再扩张" —— MVP先行，快速迭代
   - "平台化思维，开放生态" —— 做基础设施，让别人在上面做业务

3. **设计至少2种方案**

   对每种方案，需明确：

   **方案A：{方案名称}**
   - 设计理念：
   - 核心思路：
   - 关键功能：
   - 用户体验主线：
   - 技术实现要点：
   - 开发成本预估：

   **方案B：{方案名称}**
   - 设计理念：
   - 核心思路：
   - 关键功能：
   - 用户体验主线：
   - 技术实现要点：
   - 开发成本预估：

   **（可选）方案C：{方案名称}**
   - 同上

### 第三部分：方案对比与选择

4. **用对比矩阵评估各方案**

   | 评估维度 | 方案A | 方案B | 方案C | 说明 |
   |---------|-------|-------|-------|------|
   | **用户体验** | ⭐1-5 | ⭐1-5 | ⭐1-5 | 用户用起来是否顺畅？ |
   | **开发成本** | 低/中/高 | 低/中/高 | 低/中/高 | 需要多少人力/时间？ |
   | **技术风险** | 低/中/高 | 低/中/高 | 低/中/高 | 有没有技术难点？ |
   | **商业价值** | 低/中/高 | 低/中/高 | 低/中/高 | 对商业目标的贡献？ |
   | **战略对齐** | 低/中/高 | 低/中/高 | 低/中/高 | 是否符合长期战略？ |
   | **上线速度** | 快/中/慢 | 快/中/慢 | 快/中/慢 | 多久能上线？ |
   | **可扩展性** | 低/中/高 | 低/中/高 | 低/中/高 | 未来能否扩展？ |

   > 分数只是参考，最终决策需要综合判断。

5. **每个方案的最大风险是什么？**

   | 方案 | 最大风险 | 风险等级 | 缓解措施 |
   |------|---------|---------|---------|
   | 方案A | | | |
   | 方案B | | | |

6. **推荐方案与选择逻辑**

   **常见选择逻辑**：
   - 资源有限 → 选成本低的
   - 需要快速验证 → 选上线快的
   - 战略意义重大 → 选商业价值高的
   - 技术债务严重 → 选技术风险低的

7. **MVP范围定义**

   无论选哪个方案，都应定义 MVP（最小可行产品）：
   - **第一期（MVP）做**：验证核心假设所需的最小功能集
   - **第二期做**：MVP 验证后的扩展功能
   - **第三期做**：长期愿景功能

## 虚拟角色审核：技术视角

方案设计文档完成后，加载 `references/reviewers/reviewer-tech.md`，以技术视角审核，重点：

- 方案的技术可行性如何？有没有隐藏的技术难点？
- 开发成本预估是否合理？有没有低估？
- 技术风险是否可控？有没有技术债务隐患？
- 方案的可扩展性如何？未来是否能支撑业务发展？

审核后输出 3-5 个关键问题和修改建议，由用户确认后再进入下一阶段。
