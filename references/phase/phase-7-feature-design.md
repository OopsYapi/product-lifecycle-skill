# 阶段7：功能设计 & 异常场景

> 好的功能设计不是"把功能堆上去"，而是"精心设计用户的每一次交互"。本阶段目标：**详细设计每个功能，覆盖正常流程和异常场景，确保产品在各种情况下都能优雅地处理**。

---

## AI 执行指引

进入本阶段时，按以下步骤操作：

1. **读取前置产物**：读取方案设计文档、战略规划中的路线图（确定MVP范围）
2. **逐步引导提问**：按下方「引导提问」分3个部分依次讨论
3. **每个功能必须覆盖：用户故事 + 流程 + 业务规则 + 异常场景**
4. **生成功能设计文档**：按「输出模板」生成
5. **触发审核**：加载 `references/reviewers/reviewer-tech.md`、`references/reviewers/reviewer-user.md`、`references/reviewers/reviewer-legal.md`（如涉及合规），分别审核
6. **归档**：将文档保存到 `~/.workbuddy/products/{产品名称}/07-features/`
7. **下一步**：进入 **阶段8：指标设计 & ROI测算**

---

## 本阶段目标

输出：
- **详细功能设计文档**（每个功能的用户故事、流程、界面、业务规则）
- **异常场景清单**（网络/数据/并发/权限/边界/系统/业务）
- **业务规则定义**（清晰的规则边界）

## 引导提问

### 第一部分：功能设计

1. **功能列表与优先级**

   | 功能模块 | 功能名称 | 优先级 | 用户故事 | 状态 |
   |---------|---------|--------|---------|------|
   | | | P0/P1/P2 | | 新增/迭代 |

2. **每个功能的用户故事**

   > 格式：`作为 [角色]，我希望 [行为]，以便 [价值]`

   **验收条件**（每个用户故事必须配验收条件）：
   - 给定 [前置条件]，当 [触发操作]，则 [预期结果]

3. **每个功能的流程设计**

   > 流程必须包含：正常流程 + 异常分支

   ```
   开始 → 步骤1 → 步骤2 → 步骤3 → 结束
                ↓ (异常)
              错误处理 → 结束/重试
   ```

4. **每个功能的界面设计**

   | 界面 | 元素 | 交互 | 文案 | 状态 |
   |------|------|------|------|------|
   | | | | | 默认/加载/空/错误 |

   > 必须覆盖4种状态：默认态、加载态、空态、错误态

### 第二部分：异常场景设计

5. **异常场景清单**——系统性地覆盖

   | 异常类型 | 异常场景 | 用户影响 | 处理方式 | 优先级 |
   |---------|---------|---------|---------|--------|
   | **网络异常** | 超时 | 操作无响应 | 重试+提示 | P0 |
   | **网络异常** | 断网 | 功能不可用 | 离线缓存+恢复同步 | P1 |
   | **数据异常** | 格式错误 | 功能异常 | 格式校验+错误提示 | P0 |
   | **数据异常** | 数据缺失 | 信息不完整 | 默认值+友好提示 | P1 |
   | **并发异常** | 并发冲突 | 数据不一致 | 乐观锁+冲突提示 | P0 |
   | **并发异常** | 重复提交 | 重复数据 | 幂等性设计 | P0 |
   | **权限异常** | 无权限 | 操作被拒 | 权限校验+引导 | P0 |
   | **权限异常** | 登录失效 | 操作中断 | 静默刷新+状态保持 | P1 |
   | **边界异常** | 空数据 | 界面空白 | 空状态设计+引导 | P1 |
   | **边界异常** | 超长数据 | 布局错乱 | 截断+提示 | P2 |
   | **边界异常** | 特殊字符 | 功能异常 | 过滤+转义 | P1 |
   | **系统异常** | 服务不可用 | 功能不可用 | 降级+友好错误页 | P0 |
   | **业务异常** | 规则不满足 | 操作被拒 | 业务提示+引导操作 | P0 |

6. **每个核心功能的异常场景矩阵**

   | 功能 | 可能的异常 | 处理方式 | 用户看到的提示 |
   |------|-----------|---------|--------------|
   | | | | |

### 第三部分：业务规则设计

7. **业务规则清单**——规则是产品的"法律"，必须清晰无歧义

   | 规则编号 | 业务规则 | 规则细节 | 边界 | 边界外处理 |
   |---------|---------|---------|------|----------|
   | | | | | |

   > 每条规则必须能回答：什么条件下触发？触发后做什么？例外情况怎么处理？

8. **规则的优先级与冲突处理**

   - 不同规则冲突时，哪条优先？
   - 规则变更时的通知和过渡机制？

## 虚拟角色审核

功能设计完成后，加载多个审核角色：

**技术视角**（`references/reviewers/reviewer-tech.md`）：
- 功能是否可实现？技术难点在哪？
- 异常场景覆盖是否完整？
- 业务规则是否清晰可编码？

**用户视角**（`references/reviewers/reviewer-user.md`）：
- 功能是否解决真实问题？
- 用户体验是否流畅？
- 异常提示是否友好？空状态是否有引导？

**合规视角**（`references/reviewers/reviewer-legal.md`，涉及合规时）：
- 有没有隐私风险？个人信息处理是否合规？
- 有没有法律风险？

审核后输出 3-5 个关键问题和修改建议，由用户确认后再进入下一阶段。
