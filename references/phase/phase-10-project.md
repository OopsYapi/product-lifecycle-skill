# 阶段10：项目管理 & 需求池

> 产品设计再好，不能落地也只是纸上谈兵。本阶段目标：**将产品设计转化为可执行的项目计划，建立需求池管理机制，确保产品持续迭代**。

---

## AI 执行指引

进入本阶段时，按以下步骤操作：

1. **读取前置产物**：读取功能设计文档、战略规划（路线图）、ROI测算
2. **逐步引导提问**：按下方「引导提问」分3个部分依次讨论
3. **生成项目计划和需求池**：按「输出模板」生成
4. **归档**：将文档保存到 `~/.workbuddy/products/{产品名称}/10-project/`
5. **知识归档检查**：确认所有阶段产物都已归档

---

## 本阶段目标

输出：
- **项目计划**（里程碑、任务分解、资源分配、风险管理）
- **需求池**（Backlog管理、优先级、状态流转）

## 引导提问

### 第一部分：项目计划

1. **项目里程碑**——从路线图到具体时间点

   | 里程碑 | 时间 | 交付物 | 负责人 | 成功标准 |
   |--------|------|--------|--------|---------|
   | M1：MVP完成 | | MVP产品 | | 核心功能可演示 |
   | M2：内测上线 | | 内测版本 | | 内测用户反馈良好 |
   | M3：正式上线 | | 正式版本 | | 正式用户可使用 |

2. **任务分解（WBS）**——将里程碑拆成可执行的任务

   | 里程碑 | 任务 | 子任务 | 负责人 | 工期 | 依赖 |
   |--------|------|--------|--------|------|------|
   | | | | | | |

   > 每个任务的工期建议加20%缓冲。

3. **资源分配**

   | 角色 | 人员 | 投入比例 | 时间段 |
   |------|------|---------|--------|
   | 产品 | | | |
   | 设计 | | | |
   | 前端开发 | | | |
   | 后端开发 | | | |
   | 测试 | | | |
   | 运营 | | | |

4. **风险管理**

   | 风险 | 概率 | 影响 | 缓解措施 | 负责人 |
   |------|------|------|---------|--------|
   | 需求变更 | 高 | 中 | 需求冻结机制 | 产品 |
   | 技术难点 | 中 | 高 | 技术预研+备选方案 | 技术 |
   | 人力不足 | 中 | 高 | 优先级砍需求 | 产品+技术 |
   | 上线延期 | 高 | 中 | 灰度发布+分期上线 | 项目经理 |

### 第二部分：需求池管理

5. **需求池结构**

   | 字段 | 说明 |
   |------|------|
   | 需求ID | 唯一标识 |
   | 需求名称 | 简短描述 |
   | 来源 | 用户反馈/数据/竞品/老板/stakeholder |
   | 优先级 | P0(必须)/P1(应该)/P2(可以)/P3(不做) |
   | 状态 | 待评估→排队→设计中→开发中→测试→上线→关闭 |
   | 负责人 | 谁负责推进 |
   | 预计工期 | 人天 |
   | 关联指标 | 对应哪个驱动/健康指标 |

6. **需求池管理规则**

   | 维度 | 规则 |
   |------|------|
   | 新需求准入 | 谁可以提需求？怎么提？ |
   | 优先级评审 | 谁来评？多久评一次？用什么框架？ |
   | 排期规则 | 谁来排？多久排一次？ |
   | 需求变更 | 怎么提变更？谁审批？影响怎么评估？ |
   | 需求关闭 | 什么条件算完成？ |

7. **需求池工具**

   | 工具 | 说明 | 适用团队 |
   |------|------|---------|
   | TAPD | 腾讯敏捷协作 | 腾讯系 |
   | Jira | Atlassian | 国际化团队 |
   | 飞书多维表格 | 飞书协作 | 飞书系 |
   | Notion | 笔记+协作 | 小团队 |
   | Excel | 最简单 | 任何团队 |

### 第三部分：产品开发流程

8. **开发流程定义**

   ```
   需求提出 → 需求评估 → 需求排队 → 产品设计 → 技术方案 → 开发实施 → 测试验收 → 灰度发布 → 全量上线 → 数据监控 → 迭代优化
   ```

9. **关键检查点**

   | 检查点 | 检查内容 | 通过标准 | 检查人 |
   |--------|---------|---------|--------|
   | 需求评审 | 需求是否真实、清晰 | 全员无异议 | 产品负责人 |
   | 设计评审 | 设计是否可行、体验好 | 技术+设计认可 | 设计+技术负责人 |
   | 技术评审 | 技术方案是否合理 | 无P0技术风险 | 技术负责人 |
   | 测试验收 | 功能符合设计+无P0 Bug | 测试通过 | 测试负责人 |
   | 上线评审 | 发布计划完整+回滚方案 | 运维认可 | 运维负责人 |
   | 上线复盘 | 数据是否达标+有无问题 | 核心指标达标 | 产品负责人 |
