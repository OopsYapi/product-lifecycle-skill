# 阶段6：平台架构 & 开放API

> 好的架构是产品的骨架。本阶段目标：**设计可扩展、可维护的平台架构，并定义清晰的开放API，让生态伙伴能方便地接入**。

---

## AI 执行指引

进入本阶段时，按以下步骤操作：

1. **读取前置产物**：读取方案设计文档、商业模式设计文档、产品战略规划
2. **判断是否需要本阶段**：仅平台型产品需要，非平台型产品直接跳到阶段7
3. **逐步引导提问**：按下方「引导提问」分3个部分依次讨论
4. **生成架构设计文档**：按「输出模板」生成
5. **触发审核**：加载 `references/reviewers/reviewer-tech.md`，以技术视角审核
6. **归档**：将文档保存到 `~/.workbuddy/products/{产品名称}/06-architecture/`
7. **下一步**：进入 **阶段7：功能设计 & 异常场景**

---

## 本阶段目标

输出：
- **平台架构图**（系统架构、数据架构、技术选型）
- **开放API设计**（API列表、认证方式、数据格式）
- **核心实体定义**（数据模型）
- **技术风险清单**

## 引导提问

### 第一部分：平台架构

1. **平台类型**——决定了架构的核心模式

   | 平台类型 | 架构特征 | 典型产品 |
   |---------|---------|---------|
   | 交易型 | 订单系统+支付系统+清分系统 | 电商、外卖 |
   | 内容型 | 内容生产+分发+推荐 | 视频、文章 |
   | 社交型 | 用户关系+消息系统+动态流 | 微信、微博 |
   | 工具型 | API网关+计费系统+配额管理 | 云服务、开放平台 |
   | 数据型 | 数据采集+清洗+分析+输出 | 数据平台、BI |
   | 混合型 | 以上多种组合 | |

2. **核心实体与关系**——平台的核心是"实体"和"实体之间的关系"

   | 实体名称 | 说明 | 关键字段 | 与其他实体的关系 |
   |---------|------|---------|----------------|
   | | | | |

   > 建议：先用 ER 图（文字描述）梳理实体关系，再确定数据模型。

3. **系统架构分层**

   ```
   接入层：网关 / 负载均衡 / CDN
   应用层：业务服务 / 微服务
   数据层：数据库 / 缓存 / 消息队列
   基础层：监控 / 日志 / 配置 / CI/CD
   ```

   每层用什么技术？为什么？

4. **技术选型**

   | 技术维度 | 选项 | 选择 | 理由 |
   |---------|------|------|------|
   | 前端 | React/Vue/小程序 | | |
   | 后端 | Java/Node/Python/Go | | |
   | 数据库 | MySQL/PostgreSQL/MongoDB | | |
   | 缓存 | Redis/Memcached | | |
   | 消息队列 | Kafka/RabbitMQ | | |
   | 部署 | 云服务/自建/混合 | | |
   | 监控 | Prometheus/Grafana/ELK | | |

5. **非功能性需求**

   | 需求 | 要求 | 设计方案 |
   |------|------|---------|
   | 性能 | 响应时间、并发量、QPS | |
   | 可用性 | 99.9% / 99.99% | |
   | 安全性 | 认证、授权、加密、审计 | |
   | 可扩展性 | 水平扩展能力 | |
   | 可维护性 | 监控、日志、告警、灰度发布 | |

### 第二部分：开放API设计

6. **哪些能力需要开放？开放给谁？**

   | 开放能力 | 开放对象 | 开放方式 | 调用频率预估 |
   |---------|---------|---------|-------------|
   | | | | |

   > 开放方式：API接口 / SDK / Webhook / 小程序嵌入 / 数据同步

7. **API设计原则**

   | 原则 | 说明 |
   |------|------|
   | RESTful | 标准 REST 风格，资源化 URL |
   | 版本管理 | `/api/v1/`，向前兼容 |
   | 认证授权 | OAuth 2.0 + API Key |
   | 数据格式 | JSON 统一 |
   | 错误处理 | 标准错误码 + 错误信息 + 请求ID |
   | 限流 | 按调用方分级限流 |
   | 文档 | OpenAPI/Swagger 自动生成 |

8. **API列表**

   | 模块 | API名称 | 方法 | 路径 | 说明 | 调用方 |
   |------|--------|------|------|------|-------|
   | | | | | | |

9. **数据安全策略**

   | 安全维度 | 措施 | 说明 |
   |---------|------|------|
   | 传输安全 | HTTPS + 证书固定 | |
   | 存储安全 | PII 加密存储 | |
   | 访问控制 | 认证 + 授权 + 最小权限 | |
   | 审计日志 | 全量 API 调用记录 | |
   | 数据脱敏 | PII 字段脱敏返回 | |
   | 数据隔离 | 多租户数据隔离 | |

## 虚拟角色审核：技术视角

架构设计完成后，加载 `references/reviewers/reviewer-tech.md`，以技术视角审核，重点：

- 架构能不能支撑未来3年的业务发展？
- 有没有单点故障？可用性达标吗？
- 技术选型是否合适？团队能力是否匹配？
- 开放API是否规范？是否易用？
- 数据安全是否到位？

审核后输出 3-5 个关键问题和修改建议，由用户确认后再进入下一阶段。
